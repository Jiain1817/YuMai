# 民族故事问答系统 — 项目概况与进度总结

> 本文档面向项目指导人，汇总当前项目的定位、技术实现与已完成功能，便于了解详细进度。

---

## 一、项目定位与目标

本项目是一个**民族民间故事智能问答系统**（语脉后端），主要目标包括：

- **故事数据管理**：对少数民族民间故事（如藏族、彝族等）进行结构化存储与查询。
- **智能问答**：用户选择某一故事后，可针对故事内容用自然语言提问，系统基于故事文本与检索结果给出回答。
- **检索增强（RAG）**：通过语义检索从多篇故事中找出与问题最相关的内容，再结合大模型生成答案，保证回答有据可依。

整体上，系统已完成**后端服务、数据层、RAG 检索与 AI 问答**的打通，具备可演示、可对接前端的完整能力。

## 二、技术栈概览

| 类别     | 技术选型 |
|----------|----------|
|  Web 框架 | FastAPI |
| 语言     | Python 3 |
| 数据存储 | 本地 JSON（`data/stories.json`），无数据库 |
| 向量检索 | sentence-transformers（文本向量化）+ FAISS（相似度检索） |
| 大模型   | DeepSeek API（通过 OpenAI 兼容接口调用） |
| 配置     | `.env` 配置 `OPENAI_API_KEY` 等 |

依赖库包括但不限于：`fastapi`、`uvicorn`、`pydantic`、`python-dotenv`、`openai`、`sentence-transformers`、`faiss-cpu`、`numpy` 等。项目内暂无统一 `requirements.txt`，可按上述依赖在虚拟环境中安装。

```

│   ├── routes/
│   │   ├── root.py               # GET /  健康检查

│   └── services/

├── data/

│   ├── vector_meta.json          # 脚本生成的向量元数据（story_id 映射，可选）
│   └── vector.index              # 脚本生成的 FAISS 索引（可选，见下）
├── scripts/
│   └── build_vector_index.py    # 离线构建向量与 FAISS 索引并落盘
├── tools/
│   └── word_to_json.py           # 将 Word 文档（标题/民族/原文/翻译/简介）转为 stories.json
├── tests/
│   └── agent.test.py             # agent 模块的简单调用测试
├── README.md                     # 项目说明与启动方式
├── API.md                        # 接口文档（部分与当前实现有出入，见下）
└── .env                          # 环境变量（OPENAI_API_KEY 等，不提交）
```

说明：

- **在线 RAG**：当前问答使用的向量索引由 `rag_service` 在**首次调用时**根据 `stories.json` 在内存中构建（懒加载），不依赖 `scripts/build_vector_index.py` 生成的文件。脚本可用于离线预计算或与其它流程配合。
- **故事数据格式**：每条故事支持 `id`、`title`、`ethnic`、`content`（可为 `{"zh": "正文"}` 或字符串）、`intro`；兼容旧字段 `yi_text`、`chinese_text`（在无 `content.zh` 时作回退）。

## 四、已实现功能（按模块）

### 4.1 故事数据与接口

- **数据源**：`data/stories.json`，当前为数组，每条包含：
  - 必填：`id`、`title`
  - 常用：`ethnic`（民族）、`content`（正文，支持 `{"zh": "..."}` 或字符串）、`intro`（简介）
  - 可选：`yi_text`、`chinese_text`（原文/翻译，用于兼容与回退）
- **GET /stories**  
  - 返回故事列表：`id`、`title`、`summary`（正文前 80 字摘要，支持 `content.zh`）。
- **GET /story/{story_id}**  
  - 返回单条故事：`id`、`title`、`content`（原样，可能为对象或字符串）。  
  - 找不到时返回 `{"error": "story not found"}`。

数据读取在 `stories_service` 中统一处理，支持从 `content.zh` 或其它键抽取摘要与正文，保证列表与详情均可正常使用。

### 4.2 问答接口与业务流程

- **POST /ask**  
  - 请求体：`{"story_id": 1, "question": "用户问题"}`。  
  - 流程简述：
    1. 根据 `story_id` 从 `stories.json` 取出当前故事；
    2. 若故事不存在，返回固定错误结构（含 `answer`、`source`、`rag_sources`）；
    3. 若存在，调用 `agent.ask_model(question, story)`，内部会做 RAG 检索 + 大模型调用，并返回 `answer`、`source`、`rag_sources`。

- **返回结构**（成功时）示例：
  - `answer`：模型生成的回答正文。
  - `source`：主要参考故事标题（当前为 RAG 结果中第一条）。
  - `rag_sources`：参与上下文的全部故事标题列表，便于前端展示“参考来源”。

异常时由 `app.py` 捕获并返回 `{"error": "..."}`，保证接口有统一出口。

### 4.3 RAG 检索（rag_service）

- **作用**：根据用户问题，从全部故事中按**语义相似度**检索出最相关的若干条，供后续与大模型结合生成答案。
- **实现要点**：
  - 使用 **sentence-transformers**（多语言模型 `paraphrase-multilingual-MiniLM-L12-v2`）对故事正文编码为向量；正文优先从 `content.zh` 或 `content` 字符串抽取，若无则回退到 `chinese_text` / `yi_text` / `intro`。
  - 使用 **FAISS**（IndexFlatIP，归一化内积近似余弦相似度）在内存中构建索引；索引在**首次调用检索时**根据当前 `stories.json` 懒加载构建，之后复用。
  - 接口：`search_similar_stories(question, top_k=3)`，返回最多 3 条带 `_score` 的故事字典；问答时会将**用户当前选中的故事**若不在检索结果中则强制加入上下文，保证“当前故事”一定参与回答。

这样既能在多篇故事中找相关内容，又不会忽略用户正在阅读的那一篇。

### 4.4 AI 问答（agent）

- **作用**：在 RAG 检索结果 + 当前故事的基础上，拼接上下文，调用 DeepSeek 生成回答。
- **实现要点**：
  - 系统提示词约束为“中国民间故事讲解员”，要求**严格依据给定故事内容**回答，若故事中无相关信息则回答“故事中没有提到这个信息”。
  - 上下文格式：多篇故事按【故事 1】【故事 2】… 拼接，每篇包含标题与正文（优先 `content.zh`，否则用 `yi_text` / `chinese_text` / `intro` 等拼接的 `_build_context`）。
  - 大模型：DeepSeek（`deepseek-chat`），通过 OpenAI 兼容 API + `.env` 中的 `OPENAI_API_KEY` 与 base_url 配置。
  - 返回：`answer`、`source`、`rag_sources`，与上层 `ask_service` / `app.py` 的约定一致。

版本与作者信息见 `agent.py` 文件头（如 v2.0、对接后端接口 v1.2）。

### 4.5 辅助脚本与工具

- **scripts/build_vector_index.py**  
  - 从 `stories.json` 读取故事，用 `sentence-transformers`（all-MiniLM-L6-v2）生成向量，构建 FAISS 索引（IndexFlatL2）并写入 `data/vector.index`，同时将 story_id 映射写入 `data/vector_meta.json`。  
  - 当前在线问答**不依赖**该索引，索引为运行时在内存中自建；脚本适合做离线预计算、数据分析或后续改为“读盘”方案时使用。

- **tools/word_to_json.py**  
  - 将指定文件夹内的 Word 文档（.docx）按固定格式（标题、民族、原文、翻译、简介）解析，批量生成 `stories.json` 所需结构（含 `yi_text`、`chinese_text` 等），便于资料组整理好的 Word 一键导入。

### 4.6 测试与文档

- **tests/agent.test.py**：对 `agent.ask_model` 做简单调用测试（需在合适路径下运行，或调整 import）。
- **README.md**：项目说明、技术栈、启动命令（`uvicorn backend.app:app --reload`）、Swagger/ReDoc 地址。
- **API.md**：接口说明文档；需注意当前实现与文档的差异（例如列表接口已增加 `summary`，问答返回已包含 `source`、`rag_sources`，且已接入真实模型而非 mock）。

## 五、当前数据与运行状态

- **故事数量**：`data/stories.json` 中当前为 **4 条** 故事（id 1–4），涵盖藏族、彝族等，包含标题、民族、`content.zh` 长文本与简介等，字段满足现有逻辑要求。

  - `http://127.0.0.1:8000/docs`（Swagger）
  - `http://127.0.0.1:8000/redoc`（ReDoc）
  - `GET /`、`GET /stories`、`GET /story/{id}`、`POST /ask` 均可用。
- **环境**：需配置 `.env` 中的 `OPENAI_API_KEY`（用于 DeepSeek）；首次问答会触发 RAG 索引构建，可能稍慢，之后检索会较快。

## 六、与前端/文档的对接说明

- **列表**：前端可调用 `GET /stories` 获取 `id`、`title`、`summary`，用于列表页展示。
- **详情**：`GET /story/{story_id}` 返回的 `content` 可能是对象（如 `{"zh": "..."}`），前端若只展示一段中文，可取 `content.zh` 或做兼容。
- **问答**：`POST /ask` 请求体为 `story_id` + `question`；返回的 `answer` 为回答正文，`source` / `rag_sources` 可用来展示“参考了哪些故事”。
- 建议以当前后端实际行为为准，如需可再更新 `API.md` 与前端约定（含错误码、错误体格式等）。

## 七、小结：已完成与可后续扩展

**当前已完成：**

- 故事数据的 JSON 存储与多字段兼容（含 `content.zh`、民族、简介等）。

- 基于 DeepSeek 的约束性问答（仅依据故事内容回答，并带来源信息）。
